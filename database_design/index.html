Database Design Fundamentals

Learn the principles of effective database design including normalization, ER modeling, indexing strategies, and performance optimization techniques.

Introduction to Database Design

Database design is the process of producing a detailed data model of a database. This data model contains all the needed logical and physical design choices and physical storage parameters needed to generate a design.

Good database design is crucial for building scalable, high-performance applications. A well-designed database:

  • Ensures data integrity and accuracy
  • Reduces data redundancy
  • Improves query performance
  • Simplifies application development
  • Supports future growth and changes

Database Normalization

Normalization is the process of organizing data in a database to reduce redundancy and improve data integrity. The process involves dividing large tables into smaller, related tables and defining relationships between them.

Normal Forms

The normalization process is typically done through "normal forms," each with specific requirements:

  • First Normal Form (1NF): Eliminate repeating groups, ensure atomicity
  • Second Normal Form (2NF): Remove partial dependencies
  • Third Normal Form (3NF): Remove transitive dependencies
  • Boyce-Codd Normal Form (BCNF): Stricter version of 3NF
  • Fourth Normal Form (4NF): Remove multi-valued dependencies

Example: Normalization Process

Let's consider an example of an unnormalized table and how we would normalize it:

SQL 
-- Unnormalized table (all in one)
CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_name VARCHAR(100),
    customer_email VARCHAR(100),
    product_name VARCHAR(100),
    product_category VARCHAR(50),
    order_date DATE,
    quantity INT,
    price DECIMAL(10,2)
);

-- Normalized tables
CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

CREATE TABLE products (
    product_id INT PRIMARY KEY,
    name VARCHAR(100),
    category VARCHAR(50),
    price DECIMAL(10,2)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

CREATE TABLE order_items (
    order_item_id INT PRIMARY KEY,
    order_id INT,
    product_id INT,
    quantity INT,
    FOREIGN KEY (order_id) REFERENCES orders(order_id),
    FOREIGN KEY (product_id) REFERENCES products(product_id)
);

ER Diagram Components

  • Entities: Represent real-world objects (e.g., Customer, Product)
  • Attributes: Properties of entities (e.g., customer_id, name)
  • Relationships: Associations between entities (e.g., places, contains)
  • Cardinality: Defines the numerical relationship between entities (e.g., one-to-many, many-to-many)

Relationship Types

Relationship Type Description Example
One-to-One (1:1) One instance of Entity A is associated with one instance of Entity B User and UserProfile
One-to-Many (1:N) One instance of Entity A is associated with multiple instances of Entity B Customer and Orders
Many-to-Many (M:N) Multiple instances of Entity A are associated with multiple instances of Entity B Students and Courses

Index Optimization

Indexes are special data structures that improve the speed of data retrieval operations on database tables. Proper index design is crucial for database performance.

Index Types

  • B-tree indexes: Most common type, good for range queries
  • Hash indexes: Best for equality comparisons
  • Bitmap indexes: Useful for columns with low cardinality
  • Full-text indexes: Designed for text search operations
  • Composite indexes: Indexes on multiple columns

Best Practices for Indexing

  • Index columns frequently used in WHERE clauses
  • Index columns used in JOIN operations
  • Consider composite indexes for multi-column queries
  • Avoid over-indexing as it slows down write operations
  • Regularly monitor and maintain indexes
SQL 
-- Creating indexes
CREATE INDEX idx_customers_email ON customers(email);
CREATE INDEX idx_orders_customer_date ON orders(customer_id, order_date DESC);

-- Composite index for often-queried combinations
CREATE INDEX idx_products_category_price ON products(category, price);

-- Unique index to enforce uniqueness
CREATE UNIQUE INDEX idx_users_username ON users(username);

-- Analyzing query performance
EXPLAIN ANALYZE 
SELECT * FROM orders 
WHERE customer_id = 123 
AND order_date > '2023-01-01';

Performance Optimization

Database performance optimization involves various techniques to improve the speed and efficiency of database operations.

Query Optimization

  • Use appropriate WHERE clauses to limit results
  • Avoid SELECT * - only retrieve needed columns
  • Use JOINs instead of subqueries when possible
  • Limit the use of wildcards at the beginning of LIKE patterns
  • Utilize pagination for large result sets

Database Structure Optimization

  • Choose appropriate data types for columns
  • Use vertical partitioning for wide tables
  • Consider denormalization for read-heavy workloads
  • Implement partitioning for large tables
  • Use appropriate storage engines
SQL 
-- Example of an optimized query
-- Instead of:
SELECT * FROM orders WHERE YEAR(order_date) = 2023;

-- Use:
SELECT order_id, customer_id, order_date, total_amount 
FROM orders 
WHERE order_date >= '2023-01-01' 
AND order_date < '2024-01-01';

-- Instead of:
SELECT * FROM products WHERE product_name LIKE '%apple%';

-- Use (if possible):
SELECT * FROM products WHERE product_name LIKE 'apple%';

-- Using pagination for large results
SELECT * FROM orders 
ORDER BY order_date DESC 
LIMIT 50 OFFSET 100;
